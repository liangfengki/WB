## 用户需求

用户反馈图片生成速度很慢，希望找到有效办法提升处理速度。

## 核心问题

当前系统存在6个已识别的性能瓶颈：

1. 双信号量叠加导致并发死锁式等待
2. 同一产品的多场景生成是串行的
3. 每次API调用都创建新的HTTP连接（无连接池复用）
4. 自动识别每张图都额外调用一次AI API
5. 同一产品多场景生成时原图重复base64编码
6. 并发数配置偏低（CONCURRENCY=3）

## 优化目标

在不改变生成质量的前提下，通过并发优化、连接复用、减少冗余调用等手段，将整体处理速度提升2-4倍。

## 技术栈

- 后端：Python + Flask + asyncio + aiohttp
- 图像处理：Pillow + EasyOCR
- API调用：OpenAI兼容接口（chat/completions）

## 实现方案

### 优化1：消除双信号量瓶颈（影响最大）

**问题**：`BatchProcessor` 有 `semaphore(CONCURRENCY=3)`，`SeedreamAPI` 也有 `semaphore(CONCURRENCY=3)`，两层信号量形成嵌套锁定，当3个外层任务占满时，内层信号量也被锁死，实际并发只有3。
**方案**：

- 移除 `BatchProcessor` 的外层信号量，仅保留 `SeedreamAPI` 层的信号量控制API并发
- 将 `CONCURRENCY` 默认值从3提升到5，让API层直接控制最大并发请求数
- 这样5个图片任务可以真正并行，每个任务的多场景也可以并行发起

### 优化2：多场景生成并行化（影响大）

**问题**：`batch.py:96` 多场景串行循环 `for scene_idx, scene_prompt in enumerate(scenes)`，每场景等上一次完成才发起。
**方案**：

- 将串行循环改为 `asyncio.gather` 并行发起所有场景的API调用
- API层信号量已控制总并发数，不会过载
- 预期：生成3张场景图从串行3x耗时变为并行1x耗时

### 优化3：HTTP连接池复用（影响中）

**问题**：`seedream.py:62` 每次调用 `generate_image` 都 `async with aiohttp.ClientSession()` 创建新session，每次新建TCP连接+TLS握手。
**方案**：

- 在 `SeedreamAPI.__init__` 中创建共享 `aiohttp.ClientSession`，配置连接池参数
- 在 `BatchProcessor` 生命周期结束时关闭session
- 减少每次请求约100-300ms的连接建立开销

### 优化4：原图base64编码缓存（影响小）

**问题**：同一产品多场景生成时，原图每次都重新base64编码。
**方案**：

- 在 `process_single_image` 中预先编码一次 `img_base64`，传入 `generate_image` 直接使用
- 减少重复CPU开销

### 优化5：产品识别与生成分离并行（影响中，仅自动识别模式）

**问题**：当前流程是"识别完一个→再生成"，识别和生成是串行的。
**方案**：

- 先并行识别所有产品，再并行生成所有图片
- 识别阶段用独立信号量控制并发（识别请求轻量，可设更高并发）

### 性能预期

| 优化项 | 影响场景 | 预期提升 |
| --- | --- | --- |
| 消除双信号量+提高并发 | 所有场景 | 1.5-2x |
| 多场景并行化 | 生成2+张时 | 2-3x |
| HTTP连接池 | 所有场景 | 10-20% |
| base64缓存 | 多场景生成 | 微小 |
| 识别与生成分离 | 自动识别模式 | 30-50% |


**综合预期：总体速度提升2-4倍**（取决于图片数量和场景数量）

## 实现要点

- 修改 `CONCURRENCY` 默认值为5，通过环境变量仍可调
- `SeedreamAPI` 初始化时创建共享session，新增 `close()` 方法
- `BatchProcessor.process_all` 结束时调用 `self.api.close()` 关闭session
- `ProductRecognizer` 同样使用共享session模式
- 所有并行化均通过 `asyncio.gather` 实现，由API层信号量统一控制最大并发
